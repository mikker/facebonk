Facebonk Architecture

Facebonk is one shared identity model with two local steward surfaces: a CLI and a desktop app. Both use the same core APIs and can link into the same identity.

Layers

  • core/ owns the Autobonk identity model.
  • cli/ is the terminal steward app.
  • tauri/ is the thin desktop shell.
  • bare/ is the local app backend.
  • renderer/ is the plain webview UI.

Desktop Runtime

The UI does not own keys. The local Bare host owns the real IdentityManager, and peer-to-peer state stays on-device.

Request Flow

The renderer calls Tauri commands, Rust forwards JSON RPC over a local Unix socket, and the Bare host executes Facebonk operations directly in-process.

Linking Model

  1. Create the identity in the CLI or desktop app.
  2. Keep that handler online.
  3. Create a link invite.
  4. Join from another local handler with its own storage root.
  5. Both handlers now replicate and edit the same profile.

Shared Profile

The shared profile stays intentionally small:

  • displayName
  • bio
  • avatar
  • updatedAt

profile share exports a signed profile token for preview or import flows. It is separate from the live replicated identity.

Storage Rule

Each running handler needs its own local storage directory. Two handlers can control the same identity at the same time, but they cannot both open the exact same local data dir.

Logging

The desktop app backend logs startup, request lifecycle, and uncaught failures to the terminal.

npm run dev -- -- -- --storage /tmp/facebonk-app
