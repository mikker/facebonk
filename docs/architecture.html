Facebonk Architecture

Facebonk is split into a shared core plus handler-specific frontends. The CLI exists now. A future web management tool can link into the same identity and edit the same profile.

Layout

  • core/ contains the Autobonk identity model and reusable APIs.
  • cli/ contains the current command handler and storage defaults.
  • web/ is reserved for a future management UI.

Handler model

Handlers are peers, not clients of a central service. The CLI can create the first identity, a future web handler can join through an invite, and both can then replicate and edit the same shared profile.

Share cards

profile share exports a signed profile token for preview or import flows. It is separate from the live replicated identity.

Storage rule

Each running handler needs its own local storage directory. Two handlers can control the same identity at the same time. They just cannot both open the exact same local data dir. 

node cli.js --storage /tmp/facebonk-cli serve
node cli.js --storage /tmp/facebonk-web link join <invite>
